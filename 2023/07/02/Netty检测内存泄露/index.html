Netty检测内存泄露

0. 前言

Netty是个基于NIO的异步网络编程框架。其中为人称道的点如下：
1. 基于JDK NIO包提供的工具类，封装了一套1+M+N结构的高性能线程处理框架。
2. 实现一套二叉树 + 链表结构的内存分配流程，实现高性能的内存管理。
3. 提供内存泄露检测机制以供排查。
好吧，说不下去了。 今天只想谈谈Netty是如何实现内存泄露检测的。

1. 内存泄露是啥

首先内存泄露，顾名思义，并不代表真实的物理内存出现泄露的情况，请不要想歪。
内存泄露指的是，当前程序中已经无用的内存，由于某种原因，出现了未释放或者无法释放的情况。

2. Netty中的内存泄露情况

在Netty中， JVM自动GC的功能 + Netty提供的高性能内存管理功能，这两者之间就会很容易出现一下这种情况。
1. ByteBuf对象其实已经被JVM回收了，
2. 但是Netty的内存管理机制其实并没有感知到。  
其实就如下图所示： 直接内存上的这块A区域，是被ByteBuf对象所占用的。 
只有ByteBuf对象调用release方法，这块A区域才可能被新的ByteBuf对象给重新占用。

1. ByteBuf对象被回收了，意味着程序已经不使用这个对象和这块内存了。
2. 但是ByteBuf代码逻辑上依旧占用这块内存，那么就意味着这块无用内存未被释放。
==> 因此就构成了内存泄露。

3. Netty如何检测

知识点：Reference类 + WeakReference类 + ReferenceQueue类

冷知识：当WeakReference<>T reference包装的这个T对应的对象被JVM回收后， reference这个对象就会被塞到ReferenceQueue对象中（假如有指定的话）。

那么问题来了，谁干这件事情呢？

  1. JVM GC线程清理完T后，会将reference赋值给静态变量Reference.pending。
  2. Reference类中的死循环里的tryHandlePending方法，将pending对象取出放入其ReferenceQueue对象中。

总结：通过 WeakReference<>T 类 + ReferenceQueue类， 可以将被JVM回收的T对象的信息给记录下来。

Netty是如何做的

对于每一个ByteBuf类，都会有一个WeakReference类指向它(Netty中这个类叫DefaultResourceLeak)。
这个WeakReference类中有两个关键成员对象：

  1. ReferenceQueue对象： 当任意一个DefaultResourceLeak被垃圾回收后，都会放置到该对象中。
  2. Set allLeaks对象： 记录逻辑上正在使用的所有DefaultResourceLeak，
    当任意一个ByteBuf正常调用其close()等销毁方法的时候，才会移除该DefaultResourceLeak对象的指向。

因此，当需要检测是否有内存泄露的时候，从ReferenceQueue中拿出弱引用对象DefaultResourceLeak，判断是否还在allLeaks集合中。

  1. 在：意味着泄露，打印异常栈的记录信息。
  2. 不在：意味着没有泄露。
