Hot leetcode 100-无重复字符的最长子串

题目： 给定一个字符串 s ，请你找出其中不含有重复字符的 最长子串 的长度。

示例 1:

输入: s = “abcabcbb”

输出: 3

解释: 因为无重复字符的最长子串是 “abc”，所以其长度为 3。

示例 2:

输入: s = “bbbbb”

输出: 1

解释: 因为无重复字符的最长子串是 “b”，所以其长度为 1。

示例 3:

输入: s = “pwwkew”

输出: 3

解释: 因为无重复字符的最长子串是 “wke”，所以其长度为 3。
  请注意，你的答案必须是 子串 的长度，”pwke” 是一个子序列，不是子串。

分析：

双指针的思路：

  1. 右指针正常往右走，左指针动态调整位置，随时记录左右指针之间的差值的最大值。
  2. 左指针只会向右不会再向左。
  3. 搞个Map<Character, Integer>记录任意一个字符，最大的索引位置. 帮助左指针定位

Netty检测内存泄露

0. 前言

Netty是个基于NIO的异步网络编程框架。其中为人称道的点如下：
1. 基于JDK NIO包提供的工具类，封装了一套1+M+N结构的高性能线程处理框架。
2. 实现一套二叉树 + 链表结构的内存分配流程，实现高性能的内存管理。
3. 提供内存泄露检测机制以供排查。
好吧，说不下去了。 今天只想谈谈Netty是如何实现内存泄露检测的。

1. 内存泄露是啥

首先内存泄露，顾名思义，并不代表真实的物理内存出现泄露的情况，请不要想歪。
内存泄露指的是，当前程序中已经无用的内存，由于某种原因，出现了未释放或者无法释放的情况。

2. Netty中的内存泄露情况

在Netty中， JVM自动GC的功能 + Netty提供的高性能内存管理功能，这两者之间就会很容易出现一下这种情况。
1. ByteBuf对象其实已经被JVM回收了，
2. 但是Netty的内存管理机制其实并没有感知到。  
其实就如下图所示： 直接内存上的这块A区域，是被ByteBuf对象所占用的。 
只有ByteBuf对象调用release方法，这块A区域才可能被新的ByteBuf对象给重新占用。

1. ByteBuf对象被回收了，意味着程序已经不使用这个对象和这块内存了。
2. 但是ByteBuf代码逻辑上依旧占用这块内存，那么就意味着这块无用内存未被释放。
==> 因此就构成了内存泄露。

3. Netty如何检测

知识点：Reference类 + WeakReference类 + ReferenceQueue类

冷知识：当WeakReference<>T reference包装的这个T对应的对象被JVM回收后， reference这个对象就会被塞到ReferenceQueue对象中（假如有指定的话）。

那么问题来了，谁干这件事情呢？

  1. JVM GC线程清理完T后，会将reference赋值给静态变量Reference.pending。
  2. Reference类中的死循环里的tryHandlePending方法，将pending对象取出放入其ReferenceQueue对象中。

总结：通过 WeakReference<>T 类 + ReferenceQueue类， 可以将被JVM回收的T对象的信息给记录下来。

Netty是如何做的

对于每一个ByteBuf类，都会有一个WeakReference类指向它(Netty中这个类叫DefaultResourceLeak)。
这个WeakReference类中有两个关键成员对象：

  1. ReferenceQueue对象： 当任意一个DefaultResourceLeak被垃圾回收后，都会放置到该对象中。
  2. Set allLeaks对象： 记录逻辑上正在使用的所有DefaultResourceLeak，
    当任意一个ByteBuf正常调用其close()等销毁方法的时候，才会移除该DefaultResourceLeak对象的指向。

因此，当需要检测是否有内存泄露的时候，从ReferenceQueue中拿出弱引用对象DefaultResourceLeak，判断是否还在allLeaks集合中。

  1. 在：意味着泄露，打印异常栈的记录信息。
  2. 不在：意味着没有泄露。

Redis内存极致压缩

0. 前言

从标题可以看出，这大概率又是一道面试题。

面试官： 我们这边Redis内存不够了，但是我这边老板又不出扩内存，你有啥办法压缩内存，提高整体性能？

大概有以下几种方式：

1. 特殊业务类型数据

(1). 对于一些特殊场景的业务数据，比如省份，完全可以1-北京市 2-上海市 3-浙江省, 用数字代替文字。
项目工程本地或者本地缓存，存对应的映射关系。 这一定程度上能减少缓存。

(2). 很多时候，序列化反序列化会使用JSON存储。 对于这种数据，尽量用@JsonProperty来将字段名换成短名。
比如requestInfo字段，换成短名T1这样。 也能一定程度上减小缓存大小。

2. 通用压缩算法

由于Redis存储的时候，也是二进制的。完全可以用以下这些压缩算法。
(1). ProtoBuf, 序列化后会生成二进制文件。 比 XML 更小（3 ~ 10倍）、更快（20 ~ 100倍）
(2). zstd， faceBook开源的压缩算法

3. Redis自带的压缩数据结构（64位）

Redis提供的几种数据结构中，列表和哈希表结构。其底层的实现，可能会是由压缩列表来实现的。

以列表举例，正常列表的结构是双向链表，针对其中的一个节点来说，他会拥有如下结构：

  1. 左指针 8字节
  2. 右指针 8字节
  3. 数据 （数据的长度，压缩列表有要求）

如果开启压缩列表，数据长度要求小于64字节，节点个数小于1024。（虽然可以配置）。
其结构可以说，直接省去左右指针，会多一个前一项长度的字段（8bit），以及一些其他头尾节点记录总字节数和结束标志的字段。

所以近似估计，每个节点的数据算他64字节，那么使用压缩列表，相当于节省了1/5的空间。
当然压缩列表由于节点个数限制，但是可以采用拆分成多个列表的想法，每当列表长度到达1024的时候，生成新的列表去存储即可。
当然这样会增加数据处理的复杂性，但是做成组件通用也是可以接受的。

4. 总结

对于Redis内存不够用的场景，以上减少内存的方案，有些看似可能增加了数据处理的复杂度，对于Redis的操作也多了序列化和反序列化以及一些计算的耗时，
但是对于内存满载的场景，无疑是很好的方案，而且反而能加快Redis指令的执行。

因为内存降下来了， 所以一些持久化进程的压力也下来了， 可能出现的Redis主动淘汰Key的可能性也下来了， 反而会让Redis指令执行加快。

Redis访问突然变慢了

0. 前言

从标题可以看出，这大概率是一道面试题。

面试官： Redis访问的耗时突然增加了， 如何快速定位到问题所在？

我: … … 问下运维就知道了。

GG

0. 分析流程

分析Redis访问耗时变慢，从计算机硬件方面的表现来说，可能会有两种场景：

  1. CPU负载上升
  2. 网络波动

对于网络波动方面，是真的要找运维或者其他同事，帮忙登录Redis的机器看一下具体情况。
而对于CPU负载上升，很大概率由两种情况导致：

  1. Redis内存不够
  2. Redis存在大量慢指令

1. Redis内存不够

当Redis内存不够时，会发生什么，因而导致CPU负载上升，让部分业务语句执行变慢呢？

1. 主动淘汰机制频繁触发

众所周知，Redis对于内存淘汰策略，分以下几种情况：

1. 定期淘汰： Redis把有过期时间的Key放在了一个独立的字典中，定时轮询，查看Redis的过期时间到了就删除。
2. 被动淘汰： 每次执行业务指令的时候，一旦发现Key已经过期，将Key删除后再返回业务结果。
3. 主动淘汰： 当使用的内存大于maxMemory的时候，就会触发主动淘汰。
PS: 主动淘汰的算法多种多样。无差别Key随机/LRU移除，对于过期时间key的随机/LRU移除，或者直接返给业务Error（一般不会）。

所以，当内存不足时，会触发内存的主动淘汰，这会增加CPU的负担，增加Redis指令的执行耗时。

2. 持久化功能拖累

总所周知，Redis提供了持久化的功能。分以下两种

  1. RDB持久化（全量备份）： Redis会在指定的间隔时间中，将内存中的数据集快照写入磁盘。
    实际上是fork子线程，先将数据集写入临时文件，然后将文件进行替换。
  2. AOF持久化（增量备份）: Redis会将新增的Redis指令追加到AOF文件末尾。

这两种策略都各有优劣，

  1. RDB启动快，恢复文件快，但是持久化所占用的时间长，且存在时间间隔数据容易丢失。
  2. AOF恢复文件慢，但是持久化快，且数据保存更为新鲜。

当内存文件过大时，RBD策略很可能导致持久化时间过长，让CPU负载过高，导致Redis指令的执行耗时正常。

但是现在流行使用的策略是AOF，因为后面Redis某个版本引入了AOF重写的功能，极大的改善了恢复慢的问题。
但是同时也带来了可能存在的持久化耗时占用问题：

AOF重写机制：多个命令可以依据最终的内存结果，转化为少量甚至单个命令。即add delete同一个key，最后就是误操作。
而其触发的方式有两种：
1. 手动触发：bgrewriteaof指令，能手动触发。
2. 自动触发：当AOF文件体积到达配置值时等规则。

因此，当AOF重写被触发时，也会fork子进程去做AOF重写工作，这也是一个耗时很长的工作。
当CPU繁忙时，内存不够又触发该工作，无疑会拖累业务指令的执行。这也是需要去排查的问题。

2. 总结

对于Redis内存数据库来说，出现业务指令耗时增长的问题，大概率原因在于内存不足的情况。
当然也有可能是：

  1. Redis指令使用不当，扫描大量key的语句拖累了整体性能。
  2. 在业务高峰期，手动触发如bgrewriteaof指令，导致业务指令阻塞。
  3. 网络波动等问题

对于定位Redis的问题来说，平时做好监控是最为稳妥的方案。业务监控如下指标：

  1. 慢指令的配置（执行超过一定时间的指令，被记为满指令，写入文件并告警）
  2. Redis内存的告警 （超过服务器80%的比例，告警）
  3. 其他规则配置..
  4. checkHealth心跳配置

Java对象内存布局

0. 前言

一个Java对象在内存中的结构是怎么样的呢？ 是如下的三个部分吗？ 
1. 对象头	
2. 实例数据
3. 对齐填充
毫无疑问，这么回答绝对不能说是错的。
但是对象头中有什么呢？ 为什么需要有对象头呢？ 实例数据可以理解。 
对齐填充的目的又是什么呢？ 废话少说，下面开始解答这些问题。

1. 对象头

1.1. MarkWord (标记字段)

这部分比较有意思，根据对象的状态来区分不同的状态位，从而区分不同的存储结构。
有以下几种状态：

1. 轻量级锁      
2. 重量级锁  
3. GC标记    
4. 偏向锁   
5. 无锁状态

1.2. 实例数据

没啥好说的，存储的各种基础类型字段，以及对象指针等。

1.3. 对齐填充

HotSpot VM的自动内存管理系统要求对象起始地址必须是8字节的整数倍。
这就让所有对象形成了一个规范：

我们的大小必须是8字节的倍数，而且不得小于8字节。
不够的就填充。

然后其实因果是倒置的，为什么要这么做了？其实有以下好处和坏处：

1. 指针压缩
2. 内存管理规范
3. 浪费空间，更容易导致GC (坏处)

1.3.1 指针压缩

其实‘指针压缩’这名词取的很迷惑，其实学过内存寻址相关知识的人都了解。

CPU寻址的最小单位是byte，因此对于32位的操作系统来说（寄存器32位），最大的寻址范围是4G。

而由于规范了对象起始地址必须是8byte的整数倍，是不是意味着，对于JVM来说，寻址的最小单位变成8byte了。

因此原本4G内存的JVM，其中的对象指针大小， 现在只用0.5byte即可。（理论上是这样，但实际分配的还是4byte，因为32位JVM都分配4byte）
而对于32G内存的JVM，其中的对象指针大小是4byte刚好合适。 (如果内存再大， 4byte是不支持了)。

1.3.2. 内存管理规范

对齐的好处显而易见:

  1. 如果不对齐，那么分配空间时，对于不规则的内存碎片，就是头疼的情况。

  2. 如果是对齐，那么碎片的空间都是8的倍数，更容易计算出哪块空间适合当前对象分配。

1.3.4. 浪费空间，更容易导致GC

事物总有两面，对齐其实是种空间换时间的方法；

对齐使用了额外的空间， 造成了内存分配计算的便利。但是也分配一些无用数据占用内存，导致频繁的FullGC;

1.3.4. 对齐总结

总而言之，言而总之，对齐到8字节，是利大于弊的。

好处是内存管理规范 + 指针压缩，坏处是无用数据占内存导致GC。

但是对齐到16字节以及更大，就将坏处放得过大，明显是不合适的。

若是对齐放小到1字节，那其实就没开启字节压缩嘛。对于32位的JVM最大的寻址范围就只能是4G了。

4. 问答

回答前面的问题

4.1 为什么需要有对象头呢？

解答： 对象头可以记录一些关键的状态位数据，以及指针的信息，方便特殊处理。

4.2 对齐填充的目的又是什么呢？

解答： 指针压缩（可以让JVM用更小的指针大小，寻址更大的内存空间） + 规范内存管理（优化内存分配计算等逻辑）。
